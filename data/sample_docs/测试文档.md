1. RAG 技术概述

1.1 什么是 RAG

检索增强生成（Retrieval-Augmented Generation，简称 RAG）是一种将信息检索与
大语言模型生成能力相结合的技术架构。其核心思想是在模型生成回答之前，
先从外部知识库中检索出与问题相关的文档片段，将这些片段作为上下文提供给
大语言模型，从而生成更准确、更及时的回答。

传统的纯生成式模型存在以下局限：

  - 知识截止日期：模型训练完成后无法获取新知识
  - 幻觉问题：模型可能生成事实性错误的内容
  - 缺乏可解释性：无法确认回答的信息来源
  - 参数化知识有限：模型参数量限制了知识存储容量

RAG 通过引入外部知识检索机制，有效缓解了上述问题。当用户提出问题时，
系统首先从向量数据库中检索最相关的文档片段，将这些片段与原始问题一起
拼接成丰富的提示词（Prompt），最后交由大语言模型生成基于事实的回答。

1.2 RAG 的核心优势

与传统方法相比，RAG 具有以下显著优势：

第一，知识实时更新。只需更新外部知识库，无需重新训练模型即可让系统
掌握最新信息。这对于新闻、法规、产品文档等快速变化的领域尤为重要。

第二，降低幻觉风险。模型生成回答时有检索到的真实文档作为参考依据，
大大减少了编造事实的可能性。研究表明，RAG 可以将大模型的幻觉率降低
30%-50%。

第三，提升可解释性。RAG 系统可以明确展示回答所依据的源文档，用户可以
追溯每个结论的来源，增强了系统的可信度。

第四，成本效益高。相比微调（Fine-tuning）大量领域数据，RAG 只需构建
知识库和检索管道，计算成本更低，迭代速度更快。

1.3 RAG 的应用场景

RAG 技术已广泛应用于多个领域：

  - 企业知识库问答：将企业内部文档、制度规范、技术资料构建为知识库
  - 智能客服系统：结合产品文档和历史工单，提供精准的客户支持
  - 代码文档助手：为开发者提供基于项目文档的代码查询和解释
  - 教育辅导系统：基于教材和题库的知识库，为学生提供个性化辅导
  - 医疗辅助诊断：结合医学文献和临床指南，辅助医生决策
  - 法律文书分析：基于法律法规和判例库，提供法律咨询和文书审查

2. 系统架构设计

2.1 整体架构

一个完整的 RAG 系统通常包含以下核心组件：

  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
  │  文档加载器  │ → │  文本分块器  │ → │  Embedding   │
  │             │    │             │    │   模型       │
  └─────────────┘    └─────────────┘    └──────┬──────┘
                                               │
  ┌─────────────┐    ┌─────────────┐    ┌──────┴──────┐
  │  大语言模型  │ ← │  检索增强   │ ← │  向量数据库  │
  │   (LLM)     │    │  Prompt组装 │    │             │
  └─────────────┘    └─────────────┘    └─────────────┘

数据处理流程分为离线索引和在线查询两个阶段。

离线索引阶段：
  (1) 文档加载：从 PDF、Word、Markdown、网页等来源加载原始文档
  (2) 文本分块：将长文档分割为语义完整的文本块
  (3) 向量化：使用 Embedding 模型将文本块转换为向量
  (4) 存储索引：将向量及其对应的文本存入向量数据库

在线查询阶段：
  (1) 问题向量化：将用户问题转换为向量
  (2) 相似度检索：在向量数据库中查找最相似的 Top-K 文本块
  (3) 上下文组装：将检索到的文本块与问题拼接为 Prompt
  (4) 生成回答：大语言模型基于完整的上下文生成最终回答

2.2 数据预处理流水线

数据预处理是 RAG 系统的基础环节，直接影响后续检索和生成的质量。

文档加载模块需要支持多种格式：

  ┌──────────┬────────────────────────┬──────────────────┐
  │ 文件格式  │ 解析方案                │ 适用场景          │
  ├──────────┼────────────────────────┼──────────────────┤
  │ PDF      │ PyMuPDF / pdfplumber   │ 正式文档、报告    │
  │ DOCX     │ python-docx            │ 办公文档          │
  │ Markdown │ 正则 + BeautifulSoup   │ 技术文档、笔记    │
  │ HTML     │ BeautifulSoup          │ 网页内容          │
  │ TXT      │ 内置 open              │ 纯文本            │
  └──────────┴────────────────────────┴──────────────────┘

分块模块需要考虑以下因素：

  - 语义完整性：尽可能保持段落、章节的完整性
  - 块大小控制：根据 LLM 上下文窗口设置合适大小
  - 重叠策略：相邻块间保持一定重叠，避免信息断裂
  - 边界检测：利用标题、空行、标点等自然边界

2.3 检索模块设计

检索模块负责从向量数据库中快速找到与用户问题最相关的文档片段。

常用检索方式包括：

  稠密检索（Dense Retrieval）：
    使用神经网络将文本映射到语义向量空间，通过向量相似度检索。
    优点：能理解语义相似性，对词汇不匹配问题有较好的鲁棒性。
    缺点：需要 GPU 资源进行向量编码。

  稀疏检索（Sparse Retrieval）：
    基于 BM25 等传统词袋模型的检索方法。
    优点：计算效率高，精确匹配表现好。
    缺点：无法处理同义词和语义相似问题。

  混合检索（Hybrid Retrieval）：
    将稠密检索和稀疏检索的结果进行融合。
    优点：综合两者优势，在不同场景下都有稳定表现。

2.4 生成模块设计

生成模块以大语言模型为核心，负责基于检索结果生成最终回答。

Prompt 模板设计原则：

  - 明确角色定位：告诉模型它是什么角色（如"你是一个技术助手"）
  - 提供上下文：将检索到的文档片段作为参考信息
  - 限定回答范围：要求模型仅基于提供的信息回答
  - 引用来源：要求模型在回答中标注信息来源
  - 处理无答案情况：当检索结果不相关时，模型应告知无法回答

示例 Prompt 模板：

  ```
  你是一个智能知识库助手。请基于以下参考文档回答用户问题。
  如果你无法从文档中找到答案，请明确告知。
  请引用相关的文档内容来支持你的回答。

  参考文档：
  {retrieved_chunks}

  用户问题：{question}
  ```

3. 文档解析与加载

3.1 多格式文档解析

文档解析是 RAG 系统的第一道关卡。不同类型的文档有不同的解析方案。

PDF 解析：
  PDF 是最常见的文档格式之一，但解析难度较大。我们采用 PyMuPDF 和
  pdfplumber 两种方案互补：

  - PyMuPDF (fitz)：解析速度快，适合文本型 PDF，能保留文档结构
  - pdfplumber：解析精度高，擅长表格提取，适合报表类 PDF

  ```python
  import fitz  # PyMuPDF

  def extract_pdf_text(file_path):
      doc = fitz.open(file_path)
      text = ""
      for page in doc:
          text += page.get_text()
      doc.close()
      return text
  ```

Word 文档解析：
  使用 python-docx 库解析 .docx 文件，支持段落、表格、样式提取：

  ```python
  from docx import Document

  def extract_docx_text(file_path):
      doc = Document(file_path)
      paragraphs = [p.text for p in doc.paragraphs]
      return '
'.join(paragraphs)
  ```

Markdown 解析：
  Markdown 文档保留了大量结构信息（标题层级、代码块、列表），
  这些信息对语义分块非常有价值：

  ```python
  import re

  def extract_md_structure(text):
      headings = re.findall(r'^(#{1,6})\s+(.+)$', text, re.MULTILINE)
      code_blocks = re.findall(r'```(\w*)
(.*?)```', text, re.DOTALL)
      return {
          'headings': [(len(h), t) for h, t in headings],
          'code_blocks': code_blocks
      }
  ```

3.2 文档分块策略

分块是 RAG 系统中非常关键的一环。合适的区块大小直接影响检索效果和
生成质量。

常用的分块策略：

  (1) 固定大小分块
      最简单的策略，按固定字符数或 Token 数切分。
      优点：实现简单，块大小均匀
      缺点：可能切断语义完整的段落

  (2) 递归字符分块
      按优先级从高到低使用分隔符（段落→句子→词）递归切割。
      优点：尽可能保持语义边界
      缺点：对语言特性敏感（中文标点处理需要调整）

  (3) 语义分块
      利用文档的结构特征（标题、章节边界）进行分块。
      优点：保持完整的语义单元
      缺点：块大小可能不均匀

  (4) Token 分块
      基于 Token 计数分块，确保每块不超过 LLM 上下文窗口。
      优点：精确控制 Token 使用量
      缺点：需要 Token 编码器，计算开销略大

3.3 分块策略对比

下表对比了不同分块策略的关键特性：

  ┌──────────────────┬──────────┬──────────┬────────────┬──────────┐
  │ 策略              │ 语义完整 │ 大小均匀 │ 实现复杂度  │ 推荐场景  │
  ├──────────────────┼──────────┼──────────┼────────────┼──────────┤
  │ 固定大小分块       │   低     │   高     │    低      │ 简单文本  │
  │ 递归字符分块       │   中     │   中     │    低      │ 通用文本  │
  │ 语义分块           │   高     │   低     │    中      │ 结构化文档│
  │ Token 分块         │   中     │   高     │    中      │ LLM 输入 │
  │ Markdown 标题分块  │   高     │   低     │    低      │ MD 文档  │
  └──────────────────┴──────────┴──────────┴────────────┴──────────┘

分块参数的推荐配置：

  - chunk_size（块大小）：
      - 通用问答: 500-1000 字符
      - 详细分析: 1000-2000 字符
      - 完整文档: 2000-4000 字符

  - chunk_overlap（重叠量）：
      - 推荐 chunk_size 的 10%-20%
      - 例如: chunk_size=1000 时 overlap=150-200

4. Embedding 与向量检索

4.1 文本向量化

文本向量化是将人类语言转换为计算机可以理解的数值向量的过程。
在 RAG 系统中，Embedding 模型的质量直接影响检索效果。

常用 Embedding 模型：

  - text2vec-base-chinese：中文优化，768 维向量
  - BAAI/bge-large-zh-v1.5：中英双语，1024 维向量，MTEB 排名前列
  - moka-ai/m3e-base：中英双语，768 维，推理速度快
  - text-embedding-ada-002：OpenAI 出品，1536 维，多语言支持

选择 Embedding 模型的考量因素：

  - 语言支持：是否支持中文或多语言
  - 向量维度：维度越高通常表示能力越强，但存储和检索成本也越高
  - 最大输入长度：模型能处理的最大文本长度（通常 512-8192 token）
  - 推理性能：编码速度影响索引构建和查询延迟
  - 领域适配：在特定领域的表现是否满足需求

```python
from sentence_transformers import SentenceTransformer

# 加载 Embedding 模型
model = SentenceTransformer('BAAI/bge-large-zh-v1.5')

# 编码文本
texts = ["RAG 是一种检索增强生成技术", "向量数据库用于存储文本向量"]
embeddings = model.encode(texts)

print(f"向量维度: {embeddings.shape}")  # (2, 1024)
print(f"向量示例: {embeddings[0][:5]}...")  # 前5个维度
```

4.3 相似度计算

向量检索的核心是计算查询向量与文档向量之间的相似度。

常用相似度计算方法：

  余弦相似度（Cosine Similarity）：
    sim(A, B) = cos(θ) = (A·B) / (||A|| × ||B||)
    取值范围：[-1, 1]，越大越相似
    特点：关注向量方向而非长度，是使用最广泛的度量

  欧氏距离（Euclidean Distance）：
    d(A, B) = √(Σ(Ai - Bi)²)
    取值范围：[0, +∞)，越小越相似
    特点：直观的几何距离，对向量长度敏感

  点积相似度（Dot Product）：
    sim(A, B) = A·B = Σ(Ai × Bi)
    取值范围：(-∞, +∞)，越大越相似
    特点：计算效率最高，适合归一化后的向量

```python
import numpy as np

def cosine_similarity(query_vec, doc_vecs):
    # 计算查询向量与文档向量的余弦相似度
    query_norm = query_vec / np.linalg.norm(query_vec)
    doc_norms = doc_vecs / np.linalg.norm(doc_vecs, axis=1, keepdims=True)
    similarities = np.dot(doc_norms, query_norm)
    return similarities
```

5.2 Prompt 工程

Prompt 设计是影响 RAG 效果的关键因素之一。

高质量 RAG Prompt 的要素：

  1. 角色设定：
     明确模型的身份和任务。（如"你是一名专业的技术文档助手"）

  2. 上下文引入：
     清晰地标明检索到的参考文档的边界，使用 XML 标签或 Markdown
     引用块来区分不同来源的文档。

  3. 指令约束：
     - 仅基于提供的信息回答
     - 在不确定时明确说明不确定性
     - 引用具体的信息来源
     - 如果信息不足，引导用户进一步提问

  4. 输出格式：
     指定回答的结构，如：
     - 先给出直接答案
     - 提供详细解释
     - 标注信息来源
     - 补充相关建议

5.3 上下文管理

大语言模型的上下文窗口是有限的，因此需要合理管理。

上下文管理策略：

  - 滑动窗口：只保留最近的 K 个聊天轮次
  - 摘要压缩：对长对话进行摘要后再作为上下文
  - 相关性筛选：只保留与当前问题最相关的历史信息
  - Token 预算分配：合理分配 Token 给系统提示、检索结果、历史对话

```python
class ContextManager:
    def __init__(self, max_tokens=4096):
        self.max_tokens = max_tokens
        self.history = []

    def add_to_context(self, query, retrieved_chunks, system_prompt):
        # 估算各部分的 token 数
        system_tokens = len(system_prompt) // 2
        query_tokens = len(query) // 2

        # 剩余 Token 分配给检索结果
        remaining = self.max_tokens - system_tokens - query_tokens
        chunk_tokens = []

        for chunk in retrieved_chunks:
            ct = len(chunk['content']) // 2
            if ct <= remaining:
                chunk_tokens.append(chunk)
                remaining -= ct

        return {
            'system': system_prompt,
            'query': query,
            'context': chunk_tokens,
        }
```

6. 系统评估

6.1 评估指标体系

评估 RAG 系统需要从检索质量和生成质量两个维度进行。

检索质量指标：

  - Recall@K：前 K 个结果中包含相关文档的比例
  - Precision@K：前 K 个结果中相关文档的比例
  - MRR（Mean Reciprocal Rank）：第一个相关结果排名的倒数均值
  - NDCG（Normalized Discounted Cumulative Gain）：考虑排名的综合指标
  - MAP（Mean Average Precision）：平均精确率的均值

生成质量指标：

  - 忠实度（Faithfulness）：生成内容是否与检索结果一致
  - 答案准确性（Answer Relevance）：回答是否切题
  - 上下文相关性（Context Relevance）：检索结果是否与问题相关
  - 有害信息率：生成内容中是否包含不当信息
  - 引用准确率：引用信息是否正确对应源文档

6.2 评估方法

RAG 系统的评估方法可以分为人工评估和自动评估两类。

人工评估：
  由标注人员对系统输出进行评分，是最可靠的评估方式，但成本较高。
  通常采用 Likert 5 分量表，从多个维度进行评分。

自动评估：
  利用 LLM 作为评判者（LLM-as-a-Judge），或使用标准测试集。
  常用工具包括：

  - RAGAS：专为 RAG 系统设计的评估框架
  - TruLens：提供 RAG 三元组评估（忠实度、答案相关性、上下文相关性）
  - LangSmith：LangChain 生态的评估平台
  - DeepEval：开源 LLM 评估框架

```python
# 使用 RAGAS 评估 RAG 系统示例
from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_relevancy,
)

# 准备评估数据
eval_dataset = {
    "question": ["什么是 RAG 技术？"],
    "answer": ["RAG 是检索增强生成技术..."],
    "contexts": [["RAG 是一种将检索与生成结合的技术..."]],
}

# 执行评估
scores = evaluate(
    eval_dataset,
    metrics=[faithfulness, answer_relevancy, context_relevancy]
)
print(scores)
```

7. 性能优化

7.1 检索优化

检索性能直接影响 RAG 系统的响应速度和用户体验。

索引优化策略：

  1. 量化压缩（Quantization）：
     将浮点向量压缩为低精度表示（FP16 → INT8），可以减少 4 倍存储
     空间，检索速度提升 2-3 倍，精度损失通常小于 1%。

  2. 索引剪枝（Pruning）：
     移除低频或不重要的向量，减小索引规模。

  3. 分区索引（Partitioning）：
     按文档类别、时间或其他维度对向量进行分区，检索时只搜索相关分区。

  4. 缓存热门结果：
     对高频查询的结果进行缓存，避免重复计算。

7.2 缓存策略

合理的缓存策略可以大幅降低系统延迟和 API 调用成本。

多层缓存架构：

  ┌─────────────────────────────────────────────┐
  │  第一层：查询结果缓存 (Redis/Memcached)      │
  │  - 缓存完整的查询-回答对                      │
  │  - TTL: 根据业务场景设置（5分钟-24小时）       │
  │  - 适合高频重复问题                          │
  ├─────────────────────────────────────────────┤
  │  第二层：Embedding 缓存                      │
  │  - 缓存文本的向量表示                         │
  │  - 避免重复调用 Embedding API                │
  │  - LRU 淘汰策略                             │
  ├─────────────────────────────────────────────┤
  │  第三层：检索结果缓存                         │
  │  - 缓存向量检索的 Top-K 结果                  │
  │  - 适用于相似问题共享上下文                    │
  └─────────────────────────────────────────────┘

7.3 混合检索

混合检索结合了稠密检索和稀疏检索的优势，是提高检索质量的有效手段。

实现方式：

  ```python
  def hybrid_search(query, dense_index, sparse_index, alpha=0.5):
      # 稠密检索得分
      dense_scores = dense_index.search(query, k=100)

      # 稀疏检索得分 (BM25)
      sparse_scores = sparse_index.search(query, k=100)

      # 分数融合
      all_doc_ids = set(dense_scores.keys()) | set(sparse_scores.keys())
      hybrid_scores = {}

      for doc_id in all_doc_ids:
          dense_score = dense_scores.get(doc_id, 0)
          sparse_score = sparse_scores.get(doc_id, 0)

          # 线性加权融合
          hybrid_scores[doc_id] = (
              alpha * dense_score + (1 - alpha) * sparse_score
          )

      # 返回 Top-K
      return sorted(hybrid_scores.items(), key=lambda x: -x[1])[:20]
  ```

  参数 alpha 控制稠密检索和稀疏检索的权重比例：
    - alpha=1.0: 纯稠密检索
    - alpha=0.0: 纯稀疏检索
    - alpha=0.5: 均衡融合（通常效果最佳）

效果评估：
  - 文档检索时间：从平均 5 分钟降至 3 秒
  - 问题解决率：从 60% 提升至 85%

9. 总结与展望

9.1 核心要点总结

本白皮书全面介绍了 RAG 系统的技术原理、架构设计和实践方法。

核心要点：

  1. RAG 通过检索外部知识增强 LLM 生成能力，有效解决知识截止、
     幻觉等问题，是构建可靠 AI 应用的关键技术。

  2. 文档解析和分块是 RAG 的基础环节。选择合适的解析工具和分块
     策略对系统效果有重要影响。语义分块在多数场景下表现最优。

  3. Embedding 模型和向量数据库的选择需要根据实际场景权衡。
     混合检索策略通常能取得最佳检索效果。

  4. Prompt 工程和上下文管理直接影响生成质量。精心设计 Prompt
     模板可以显著提升系统效果。

  5. 系统评估需要同时关注检索质量和生成质量。自动化评估工具如
     RAGAS 可以大幅降低评估成本。

  6. 通过缓存、量化、索引优化等手段，可以有效提升系统性能。

9.2 未来发展方向

RAG 技术仍在快速发展中，未来值得关注的方向包括：

  1. 多模态 RAG：
     将图像、表格、音频等多模态信息纳入检索和生成过程。

  2. Agentic RAG：
     让 RAG 系统具备主动推理和工具使用能力，能够进行复杂的多步
     推理任务。

  3. 自适应 RAG：
     根据问题的复杂度和类型，动态调整检索策略和参数。

  4. 图增强 RAG (GraphRAG)：
     结合知识图谱的结构化信息，增强对实体关系的理解和推理。

  5. 流式 RAG：
     支持实时数据流的检索和生成，适用于新闻、金融等时效性要求
     高的场景。

  6. 隐私保护 RAG：
     在检索和生成过程中保护数据隐私，支持联邦学习和差分隐私。

基于实践经验，以下是构建 RAG 系统的最佳实践建议：
